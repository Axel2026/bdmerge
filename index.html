TEMAT : Zapytania wybierające z klauzulą WHERE

  1. Wprowadzenie

    1. Definiowanie zapytania wybierającego z klauzulą WHERE.

      Klauzula WHERE jest klauzulą selekcji, która wybiera z tabeli wszystkie rekordy spełniające określony warunek. Przy definiowaniu kryteriów wyboru rekordów z tabeli można wykorzystać następujące klasy predykatów:

      Klasa predykatów Alternatywna notacja
      		 Opis
      =, <>, >, >=, <, <=
      		Porównuje dwie wartości, dwa wyrażenia
      BETWEEN...AND...
      		Sprawdza czy wartość znajduje się w określonym zakresie (przedziale domkniętym)
      IN(...,...)
      		Sprawdza czy wartość znajduje się na zadanej liście (w zadanym zbiorze)
      LIKE
      NOT LIKE
      		 ~~
      !~~
      		 Sprawdza czy wartość odpowiada podanemu wzorcowi
      (% - oznacza dowolny łańcuch znaków /także zerowy/,
      _ - oznacza dowolny, dokładnie jeden znak)
      ILIKE
      NOT ILIKE
      		 ~~*
      !~~*
      		 j.w. (nie jest brana pod uwagę wielkość liter)
      SIMILAR TO
      NOT SIMILAR TO
      		Sprawdza czy wartość odpowiada podanemu wzorcowi
      (% - oznacza dowolny łańcuch znaków /także zerowy/,
      _ - oznacza dowolny, dokładnie jeden znak,
      | - oznacza alternatywę,
      * - oznacza powtórzenie poprzedniego znaku 0 lub więcej razy,
      + - oznacza powtórzenie poprzedniego znaku 1 lub więcej razy)
      W odróżnieniu od predykatu LIKE/ILIKE może wykorzystywać wyrażenia regularne
      ~
      !~
      		Sprawdza czy wartość odpowiada podanemu wzorcowi
      (. - oznacza dowolny znak,
      | - oznacza alternatywę,
      * - oznacza powtórzenie poprzedniego znaku 0 lub więcej razy,
      + - oznacza powtórzenie poprzedniego znaku 1 lub więcej razy)
      W odróżnieniu od predykatu SIMILAR TO może wykorzystywać wyrażenia regularne standardu POSIX
      ~*
      !~*
      		j.w. (nie jest brana pod uwagę wielkość liter)
      IS NULL
      IS NOT NULL
      		Sprawdza czy wartość jest wartością NULL
      Sprawdza czy wartość nie jest wartością NULL

      Składnia:

      SELECT [nazwa_kolumny AS alias_kolumny,...,nazwa_kolumny AS alias_kolumny | *]

      		FROM nazwa_tabeli

      		WHERE warunek;

      Przykłady:

      a) Operatory LIKE (~~), SIMILAR TO oraz wyrażenia regularne POSIX.

      b) Operatory BETWEN .. AND, IN, IS oraz OVERLAPS.

      3. Klauzula WHERE może zawierać operatory logiczne OR, AND oraz NOT.

      A OR B TRUE FALSE NULL
      TRUE TRUE TRUE TRUE
      FALSE TRUE FALSE NULL
      NULL TRUE NULL NULL

      A AND B TRUE FALSE NULL
      TRUE TRUE FALSE NULL
      FALSE FALSE FALSE FALSE
      NULL NULL FALSE NULL

      A NOT A
      TRUE FALSE
      FALSE TRUE

    3. Definiowanie zapytania wybierającego z klauzulą ORDER BY.

      Klauzula ORDER BY określa kolejność rekordów w jakiej zostaną zwrócone jako rezultat zapytania. Rekordy wynikowe mogą być posortowane w następujący sposób:

      • rosnąco ASC (ang. ascending);
      • malejąco DESC (ang. descending).

      Składnia:

      SELECT [nazwa_kolumny AS alias_kolumny,...,nazwa_kolumny AS alias_kolumny | *]

      		FROM nazwa_tabeli

      		ORDER BY nazwa_kolumny [ASC | DESC] [NULLS {FIRST | LAST}],...,nazwa_kolumny [ASC | DESC] [NULLS {FIRST | LAST}];

      SELECT [nazwa_kolumny AS alias_kolumny,...,nazwa_kolumny AS alias_kolumny | *]

      		FROM nazwa_tabeli

      		ORDER BY nr_kolumny [ASC | DESC] [NULLS {FIRST | LAST}],...,nr_kolumny [ASC | DESC] [NULLS {FIRST | LAST}];

      Przykłady:

      SELECT *

      		FROM dziekanat.studenci

      		ORDER BY nazwisko, imie ASC;

      SELECT *

      		FROM dziekanat.studenci

      		ORDER BY 3, 2 ASC;

      SELECT nr_albumu, imie, nazwisko

      		FROM dziekanat.studenci

      		ORDER BY 1 ASC;

      SELECT nr_albumu::CHARACTER(5), imie, nazwisko

      		FROM dziekanat.studenci

      		ORDER BY 1 ASC;

      SELECT lpad(nr_albumu, 5, '0')::CHARACTER(5), imie, nazwisko

      		FROM dziekanat.studenci

      		ORDER BY 1 ASC;

      SELECT nr_albumu, imie, nazwisko

      		FROM dziekanat.studenci

      		ORDER BY lpad(nr_albumu, 5, '0')::CHARACTER(5) ASC;

      SELECT nr_albumu::INTEGER, imie, nazwisko

      		FROM dziekanat.studenci

      		ORDER BY 1 ASC;

      SELECT nr_albumu, imie, nazwisko

      		FROM dziekanat.studenci

      		ORDER BY nr_albumu::INTEGER ASC;



    4. Definiowanie zapytania wybierającego z klauzulami ALL oraz DISTINCT.

      Klauzula DISTINCT stosowana jest w celu wyeliminowania powtarzających się rekordów.

      Składnia:

      SELECT [ALL | DISTINCT [ON klauzula_nazwa_kolumny_1,..., klauzula_nazwa_kolumny_m]] [nazwa_kolumny_1 AS alias_kolumny_1,...,nazwa_kolumny_n AS alias_kolumny_n | *]

      		FROM nazwa_tabeli;

      Przykłady:

      SELECT DISTINCT wku AS "WKU"

      		FROM dziekanat.wojsko;

      SELECT ALL wku AS "WKU"

      		FROM dziekanat.wojsko;

      SELECT DISTINCT tryb_studiow, stopien_studiow

      		FROM dziekanat.kierunki_studiow

      		ORDER BY 1 DESC, 2;

      SELECT DISTINCT ON (tryb_studiow, stopien_studiow) tryb_studiow, stopien_studiow

      		FROM dziekanat.kierunki_studiow

      		ORDER BY 1 DESC, 2;

      SELECT DISTINCT ON (tryb_studiow) tryb_studiow, stopien_studiow

      		FROM dziekanat.kierunki_studiow

      		ORDER BY 1 DESC, 2;

      SELECT DISTINCT ON (tryb_studiow, stopien_studiow) tryb_studiow

      		FROM dziekanat.kierunki_studiow

      		ORDER BY 1 DESC;

      .:: Intencjonalne pomijanie danych przy użyciu klauzuli DISTINCT ON ::.

      W przypadku użycia klauzuli DISTINCT ON może nastąpić pominięcie określonych danych.

      Przykład:
      1) Poniższe zapytanie zwraca wynagrodzenia wszystkich prowadzacych posortowane rosnąco wg stanowiska oraz malejąco wg płacy zasadniczej z wartościami NULL umieszczonymi na końcu. Rezultat może zawierać duplikaty.
      SELECT stanowisko, placa_zasadnicza
      FROM kadry.prowadzacy
      ORDER BY 1, 2 DESC NULLS LAST;

      2) Stosując klauzulę DISTINCT ON możliwe jest wybranie spośród wszystkich rekordów zwróconych przez w/w zapytanie (1), tylko tych, które znajdują się na początku każdej grupy stanowisk. W rezultacie zapytanie zwraca najwyższe wynagrodzenia na poszczególnych stanowiskach.
      SELECT DISTINCT ON (stanowisko) stanowisko, placa_zasadnicza
      FROM kadry.prowadzacy
      ORDER BY 1, 2 DESC NULLS LAST;

TEMAT : Zapytania z operatorami UNION, INTERSECT, EXCEPT

  1. Wprowadzenie

    1. Definiowanie łączenia tabel z wykorzystaniem operatora UNION.

      Klauzula CORRESPONDING BY nie jest dla operatora UNION wspierana przez PostgreSQL.

      Składnia:

      SELECT [nazwa_kolumny AS alias_kolumny,...,nazwa_kolumny AS alias_kolumny | *]

      		FROM nazwa_schematu_1.nazwa_tabeli_1
      UNION [ALL | CORRESPONDING [BY [nazwa_kolumny,...,nazwa_kolumny]]]
      SELECT [nazwa_kolumny AS alias_kolumny,...,nazwa_kolumny AS alias_kolumny | *]

      		FROM nazwa_schematu_2.nazwa_tabeli_2

      Przykład:

      SELECT nr_albumu

      		FROM dziekanat.studenci_kierunkow WHERE id_kierunku_studiow = 1
      UNION
      SELECT nr_albumu

      		FROM dziekanat.studenci_kierunkow WHERE id_kierunku_studiow = 5;

      SELECT nr_albumu

      		FROM dziekanat.studenci_kierunkow WHERE id_kierunku_studiow = 1
      UNION ALL
      SELECT nr_albumu

      		FROM dziekanat.studenci_kierunkow WHERE id_kierunku_studiow = 5;


    2. Definiowanie łączenia tabel z wykorzystaniem operatora INTERSECT.

      Operator INTERSECT realizuje przecięcie dwóch tabel, którego rezultatem jest tabela wynikowa zawierająca wszystkie rekordy, które są wspólne dla obu tabel.

      Klauzula CORRESPONDING BY nie jest dla operatora INTERSECT wspierana przez PostgreSQL.

      Składnia:

      SELECT [nazwa_kolumny AS alias_kolumny,...,nazwa_kolumny AS alias_kolumny | *]

      		FROM nazwa_schematu_1.nazwa_tabeli_1
      INTERSECT [ALL | CORRESPONDING [BY [nazwa_kolumny,...,nazwa_kolumny]]]
      SELECT [nazwa_kolumny AS alias_kolumny,...,nazwa_kolumny AS alias_kolumny | *]

      		FROM nazwa_schematu_2.nazwa_tabeli_2

      Przykład:

      SELECT nr_albumu

      		FROM dziekanat.studenci_kierunkow WHERE id_kierunku_studiow = 1
      INTERSECT
      SELECT nr_albumu

      		FROM dziekanat.studenci_kierunkow WHERE id_kierunku_studiow = 5;


    3. Definiowanie łączenia tabel z wykorzystaniem operatora EXCEPT.

      Operator EXCEPT realizuje różnicę dwóch tabel, której rezultatem jest tabela wynikowa zawierająca wszystkie rekordy tabeli pierwszej nie występujące w tabeli drugiej.

      Klauzula CORRESPONDING BY nie jest dla operatora EXCEPT wspierana przez PostgreSQL.

Klauzula DISTINCT definiuje, że przy obliczaniu funkcji zostaną uwzględnione tylko jednokrotne tzn. unikalne wystąpienia wartości rekordów. Natomiast klauzula ALL uwzględnia wszystkie, tzn. także powtarzające się wystąpienia wartości rekordów.

        Funkcja Opis
        SUM([DISTINCT | ALL] wyrażenie) Suma wartości wyrażenia
        AVG([DISTINCT | ALL] wyrażenie) Średnia ze wszystkich wartości wyrażenia
        MIN([DISTINCT | ALL] wyrażenie) Wartość najmniejsza wyrażenia
        MAX([DISTINCT | ALL] wyrażenie) Wartość największa wyrażenia
        COUNT([DISTINCT | ALL] wyrażenie) Liczba wystąpień wartości w wyrażeniu
        STDDEV([DISTINCT | ALL] wyrażenie) Odchylenie standardowe dla populacji zdefiniowanej przez wyrażenie
        VARIANCE([DISTINCT | ALL] wyrażenie) Wariancja dla populacji definiowanej przez wyrażenie

      2. Zapoznaj się ze sposobem definiowania zapytania wybierającego z funkcją agregującą.

        Składnia:

        SELECT funkcja([DISTINCT | ALL] nazwa_kolumny) AS alias_kolumny,...,funkcja(nazwa_kolumny) AS alias_kolumny

        		FROM nazwa_schematu.nazwa_tabeli

        		[WHERE warunek]

        		[GROUP BY nazwa_kolumny,...,nazwa_kolumny [HAVING warunek]]

        		[ORDER BY nazwa_kolumny [ASC | DESC]];

        Przykłady:

        SELECT COUNT(nr_albumu) AS "Liczba studentów"

        		FROM dziekanat.studenci;

        SELECT AVG(placa_zasadnicza) AS "Średnie wynagrodzenie"

        		FROM kadry.prowadzacy;

        SELECT AVG(ocena) AS "Średnia ocen", STDDEV(ocena) AS "Odchylenie standardowe ocen"

        		FROM dziekanat.oceny NATURAL JOIN dziekanat.przedmioty

        		WHERE nazwa_przedmiotu LIKE 'Anatomia';

        SELECT COUNT(imie) AS "Liczba imion", COUNT(DISTINCT imie) AS "Liczba imion unikalnych"

    TEMAT : Definiowanie podzapytań.

    1. Wprowadzenie.

      Podzapytanie (zagnieżdżone zapytanie) jest to polecenie SELECT umieszczone w obrębie innego polecenia SQL, na przykład polecenia DELETE, UPDATE, INSERT lub SELECT.

      Definiując podzapytanie, konieczne jest zachowanie następujących reguł:

      • Podzapytanie zagnieżdżone w zapytaniu SELECT musi zwracać:
        • pojedynczą wartość (przykład 1, 2)
        • pojedynczą kolumnę wartości (przykład 3, 4)
        • zbiór rekordów (przykład 5)
      • Podzapytanie może odwoływać się do tabel wymienionych po klauzuli FROM zapytania obejmującego, nadrzędnego (przykład 2);
      • Podzapytanie nie może zawierać klauzuli ORDER BY;
      • Podzapytanie może zawierać tylko jedno polecenie SELECT, zatem nie można w podzapytaniach wykorzystywać operatorów UNION, EXCEPT, INTERSECT.

      Przykłady:

      SELECT s.imie, s.nazwisko, s.pesel,

      		(SELECT wku FROM dziekanat.wojsko w WHERE w.nr_albumu = s.nr_albumu) AS wku
      FROM dziekanat.studenci s
      WHERE s.nr_albumu = '15';

      SELECT s.nr_albumu AS album, s.imie, s.nazwisko

      		FROM dziekanat.studenci s

      		WHERE (SELECT AVG(o.ocena) FROM dziekanat.oceny o WHERE o.nr_albumu = s.nr_albumu) > (SELECT AVG(ocena) FROM dziekanat.oceny);

      SELECT nazwa_fakultetu FROM dziekanat.fakultety

      		WHERE id_fakultetu NOT IN


      		(SELECT id_fakultetu FROM dziekanat.zapisy);

      DELETE FROM dziekanat.fakultety

      		WHERE id_fakultetu NOT IN


      		(SELECT(id_fakultetu) FROM dziekanat.zapisy WHERE data_czas_zapisu > CURRENT_DATE - INTERVAL '5 year');

      SELECT studentki.imie, studentki.nazwisko, studentki.data_urodzenia
      FROM (SELECT * FROM dziekanat.studenci WHERE pesel SIMILAR TO '%(0|2|4|6|8)_') AS studentki
      WHERE data_urodzenia BETWEEN '1992-01-01' AND '1992-12-31';

    1. Wprowadzenie.

      1. Zapoznaj się z wykorzystaniem operatora EXISTS w podzapytaniach.

        Predykat EXISTS jest zawsze wykorzystywany w połączeniu z podzapytaniem w celu sprawdzenia, czy istnieje przynajmniej jeden rekord spełniakący kryteria określone w podzapytaniu. Jeżeli podzapytanie zwraca co najmniej jedną wartość, spełniającą kryteria wyboru, predykat EXISTS ewaluuje podzapytanie na wartość logiczną TRUE, w przeciwnym przypadku ewaluuje podzapytanie na wartość FALSE.

        SELECT [nazwa_kolumny AS alias_kolumny,...,nazwa_kolumny AS alias_kolumny | *]

        		WHERE [[NOT] EXISTS] podzapytanie;

        Przykłady:

        SELECT nr_albumu, imie, nazwisko

        		FROM dziekanat.studenci s

        		WHERE NOT EXISTS


        		(SELECT * FROM dziekanat.zapisy z WHERE z.nr_albumu = s.nr_albumu);

        SELECT nr_albumu AS "ALBUM", imie AS "IMIĘ", nazwisko AS "NAZWISKO"

        		FROM dziekanat.studenci s

        		WHERE EXISTS


        		(SELECT nr_albumu, id_przedmiotu FROM dziekanat.oceny o GROUP BY nr_albumu, id_przedmiotu HAVING COUNT(ocena) = 3 AND o.nr_albumu = s.nr_albumu);

Warunkiem poprawnego fukcjonowania zapytania jest użycie przed argumentem prawostronnym predykatu ANY lub ALL.

        • w przypadku predykatu ALL, wszystkie wartości znajdujące się po prawej stronie operatora porównania spełniają warunek;

        		WHERE [wyrażenie | podzapytanie] [=|<>|>|>=|<|<=] [ANY | ALL] [wyrażenie | podzapytanie];

        SELECT nr_albumu, imie, nazwisko

        		FROM dziekanat.studenci s

        		WHERE (SELECT COUNT(*) FROM dziekanat.zapisy z WHERE z.nr_albumu = s.nr_albumu) >= ALL

    1. Wprowadzenie.

      Podzapytania mogą być zagnieżdżone tworząc tzw. zapytania wielopoziomowe (2-, 3-, 4- i więcej poziomowe). Liczba poziomów zagnieżdżeń, w standardzie SQL nie jest ograniczona, natomiast w rzeczywistych SZBD liczba ta jest ograniczona do kilkunastu poziomów.

      SELECT o.nr_albumu AS "Nr albumu", o.telefon_stacjonarny AS "Telefon stacjonarny", o.telefon_komorkowy AS "Telefon komorkowy", o.email AS "E-mail",


      		(SELECT (SELECT a.ulica || ' ' || a.nr_domu || ', ' || a.kod_pocztowy || ' ' || a.miejscowosc



      		FROM dziekanat.adresy a WHERE a.id_adresu = s.adres_korespondencyjny)


      		FROM dziekanat.studenci s WHERE s.nr_albumu = o.nr_albumu) AS "Adres korespondencyjny"

      		FROM dziekanat.osobiste o

      		WHERE o.stopien_niepelnosprawnosci IS NOT NULL;

    TEMAT : Definiowanie struktury tabeli.

    1. Wprowadzenie.

      1. Definiowanie struktury tabeli bez więzów integralności.

        Składnia:

        CREATE TABLE nazwa_schematu.nazwa_tabeli (

        		nazwa_kolumny typ_danych,

        		...

        		nazwa_kolumny typ_danych
        );

        Przykład:

        CREATE TABLE roznosci.studenci_1 (

        		nr_albumu CHARACTER VARYING(6),

        		imie CHARACTER VARYING(20),

        		nazwisko CHARACTER VARYING(30),

        		data_urodzenia DATE,

        		kierunek_studiow CHARACTER VARYING(50),

        		adres_zameldowania INTEGER
        );

      2. Definiowanie struktury tabeli z klauzulą DEFAULT.

        Klauzula DEFAULT określa wartość domyślną pola kolumny (atrybutu) tabeli. Wartość domyślna jest to wartość jaka zostanie wprowadzona w polu kolumny w przypadku gdy nie została ona określona przez użytkownika.

        Składnia:

        CREATE TABLE nazwa_schematu.nazwa_tabeli (

        		nazwa_kolumny typ_danych DEFAULT wartosc_domyslna,

        		...

        		nazwa_kolumny typ_danych DEFAULT wartosc_domyslna
        );

        Przykład:

        CREATE TABLE roznosci.studenci_2 (

        		nr_albumu CHARACTER VARYING(6),

        		imie CHARACTER VARYING(20),

        		nazwisko CHARACTER VARYING(30),

        		data_urodzenia DATE,

        		kierunek_studiow CHARACTER VARYING(50)


        		DEFAULT 'Informatyka',

        		adres_zameldowania INTEGER
        );

      3. Definiowanie struktury tabeli z ograniczeniami NULL oraz NOT NULL.

        Warunki NULL oraz NOT NULL określają, czy dopuszczamy, aby pola poszczególnych kolumn (atrybutów) tabeli mogły przyjmować wartości nieokreślone. Opcja NULL definiuje, że pola kolumny mogą przyjmować wartości nieokreślone, natomiast opcja NOT NULL przeciwnie, definiuje, że pola kolumny nie mogą przyjmować takich wartości.

        Składnia:

        CREATE TABLE roznosci.studenci_3 (

        		nr_albumu CHARACTER VARYING(6) NOT NULL,

        		imie CHARACTER VARYING(20) NOT NULL,

        		nazwisko CHARACTER VARYING(30) NOT NULL,

        		data_urodzenia DATE,

        		kierunek_studiow CHARACTER VARYING(50)


        		DEFAULT 'Informatyka',

        		adres_zameldowania INTEGER
        );

        CREATE TABLE nazwa_schematu.nazwa_tabeli (

        		nazwa_kolumny typ_danych [NULL | NOT NULL],

        		...

        		nazwa_kolumny typ_danych [NULL | NOT NULL]
        );

        Przykład:

        CREATE TABLE roznosci.przedmioty (

        		id_przedmiotu INTEGER NOT NULL,

        		nazwa_przedmiotu CHARACTER VARYING(30) NOT NULL,

        		nr_semestru SMALLINT NULL
        );

      4. Definiowanie tabeli z ograniczeniami sprawdzania poprawności CHECK.

        Klauzula CHECK definiuje warunek jaki muszą spełniać wartości pól określonej kolumny (atrybutu) tabeli. Warunek CHECK nie może odwoływać się do atrybutów innych tabel a także nie może zawierać podzapytań.

        Składnia:

        CREATE TABLE nazwa_schematu.nazwa_tabeli (

        		nazwa_kolumny typ_danych,

        		...

        		nazwa_kolumny typ_danych,

        		CONSTRAINT nazwa_ograniczenia CHECK (warunek_logiczny),

        		...

        		CONSTRAINT nazwa_ograniczenia CHECK (warunek_logiczny)
        );

        CREATE TABLE nazwa_schematu.nazwa_tabeli (

        		nazwa_kolumny typ_danych CHECK (warunek_logiczny),

        		...

        		nazwa_kolumny typ_danych CHECK (warunek_logiczny)
        );

        Przykłady:

        CREATE TABLE roznosci.oceny_1 (

        		nr_albumu INTEGER,

        		id_przedmiotu INTEGER,

        		data_zaliczenia DATE,

        		ocena REAL,

        		CONSTRAINT oceny_1_chk CHECK (ocena > 2.0 AND ocena <= 5.0)
        );

        CREATE TABLE roznosci.oceny_2 (

        		nr_albumu INTEGER,

        		id_przedmiotu INTEGER,

        		data_zaliczenia DATE,

        		ocena REAL CHECK (ocena > 2.0 AND ocena < 5.0)
        );

      5. Definiowanie struktury tabeli z więzami integralności klucza unikalnego UNIQUE.

        Klauzula UNIQUE definiuje klucz unikalny tabeli. Klucz unikalny tabeli może przyjmować wartości NULL.

        Składnia:

        CREATE TABLE nazwa_schematu.nazwa_tabeli (

        		nazwa_kolumny typ_danych,

        		...

        		nazwa_kolumny typ_danych,

        		CONSTRAINT nazwa_ograniczenia UNIQUE (nazwa_kolumny,...,nazwa_kolumny)
        );

        CREATE TABLE nazwa_schematu.nazwa_tabeli (

        		nazwa_kolumny typ_danych UNIQUE,

        		...

        		nazwa_kolumny typ_danych
        );

        Przykłady:

        CREATE TABLE roznosci.adresy_1 (

        		id_adresu INTEGER,

        		ulica CHARACTER VARYING(40),

        		nr_domu CHARACTER VARYING(5),

        		miasto CHARACTER VARYING(20),

        		kod_pocztowy CHARACTER VARYING(6),

        		wojewodztwo CHARACTER VARYING(20),

        		kraj CHARACTER VARYING(20),

        		CONSTRAINT adresy_1_uk UNIQUE (id_adresu)
        );

        CREATE TABLE roznosci.adresy_2 (

        		id_adresu INTEGER UNIQUE,

        		ulica CHARACTER VARYING(40),

        		nr_domu CHARACTER VARYING(5),

        		miasto CHARACTER VARYING(20),

        		kod_pocztowy CHARACTER VARYING(6),

        		wojewodztwo CHARACTER VARYING(20),

        		kraj CHARACTER VARYING(20)

      6. Definiowanie tabel z więzami integralności klucza głównego PRIMARY KEY.

        Warunek PRIMARY KEY definiuje klucz główny tabeli. Tabela może posiadać tylko jeden klucz główny, który może być złożony z jednego atrybutu (klucz prosty) lub z wielu atrybutów (klucz złożony). Klucz główny tabeli w odróżnieniu do klucza unikalnego nie może przyjmować wartości NULL.

        Składnia:

        CREATE TABLE nazwa_schematu.nazwa_tabeli (

        		nazwa_kolumny typ_danych,

        		...

        		nazwa_kolumny typ_danych,

        		CONSTRAINT nazwa_ograniczenia PRIMARY KEY (nazwa_kolumny,...,nazwa_kolumny)
        );

        CREATE TABLE nazwa_schematu.nazwa_tabeli (

        		nazwa_kolumny typ_danych PRIMARY KEY,

        		...

        		nazwa_kolumny typ_danych
        );

        Przykłady:

        CREATE TABLE roznosci.adresy_3 (

        		id_adresu INTEGER,

        		ulica CHARACTER VARYING(40),

        		nr_domu CHARACTER VARYING(5),

        		miasto CHARACTER VARYING(20),

        		kod_pocztowy CHARACTER VARYING(6),

        		wojewodztwo CHARACTER VARYING(20),

        		kraj CHARACTER VARYING(20),

        		CONSTRAINT adresy_3_pk PRIMARY KEY (id_adresu)
        );

        CREATE TABLE roznosci.adresy_4 (

        		id_adresu INTEGER PRIMARY KEY,

        		ulica CHARACTER VARYING(40),

        		nr_domu CHARACTER VARYING(5),

        		miasto CHARACTER VARYING(20),

        		kod_pocztowy CHARACTER VARYING(6),

        		wojewodztwo CHARACTER VARYING(20),

        		kraj CHARACTER VARYING(20)
        );

      7. Definiowanie tabel z więzami integralności klucza obcego FOREIGN KEY.

        Klauzula FOREIGN KEY definiuje klucz obcy reprezentujący związek z inną tabelą. Warunek FOREIGN KEY określa, że podczas wprowadzania lub modyfikacji danych sprawdzane będzie, czy zależności pomiędzy tabelami pozostają zachowane. Realizowane jest to w następujący sposób:

        • nie można wprowadzić wartości w polu tabeli powiązanej (tabeli z kluczem obcym), której nie odpowiada żadna wartość tabeli głównej (tabeli z kluczem kandydującym, np. kluczem głównym),
        • nie można zmieniać wartości w polu tabeli głównej, jeżeli istnieją odpowiadające mu rekordy w tabeli powiązanej (dyrektywa ON UPDATE CASCADE pozwala wyeliminować to ograniczenie, powodujące, że wraz ze zmianą wartości rekordu w tabeli głównej aktualizowane będą wszystkie powiązane z nim rekordy w tabeli powiązanej),
        • nie można usunąć rekordu z tabeli podstawowej, jeżeli istnieją odpowiadające mu rekordy w tabeli powiązanej (dyrektywa ON DELETE CASCADE umożliwia wyeliminowanie tego ograniczenia, powodując, że wraz z usunięciem rekordu w tabeli głównej usuwane będą wszystkie powiązane z nim rekordy w tabeli powiązanej).

        Składnia:

        CREATE TABLE nazwa_schematu.nazwa_tabeli_powiązanej (

        		nazwa_kolumny_tab_powiązanej typ_danych,

        		...

        		nazwa_kolumny_tab_powiązanej typ_danych,

        		CONSTRAINT nazwa_ograniczenia FOREIGN KEY (nazwa_kolumny,...,nazwa_kolumny) REFERENCES nazwa_schematu.nazwa_tabeli_głównej (nazwa_kolumny_tab_głównej,...,nazwa_kolumny_tab_głównej)
        );

        Przykłady:

        CREATE TABLE roznosci.kontakty_1 (

        		nr_albumu CHARACTER VARYING(6) UNIQUE

        		telefon_domowy CHARACTER VARYING(10),

        		telefon_komorkowy CHARACTER VARYING(10),

        		adres_email CHARACTER VARYING(50),

        		adres_korespondencyjny INTEGER,

        		CONSTRAINT kontakty_1_fk FOREIGN KEY (nr_albumu) REFERENCES roznosci.studenci_1 (nr_albumu)
        );

        CREATE TABLE roznosci.kontakty_2 (

        		nr_albumu CHARACTER VARYING(6),

        		telefon_domowy CHARACTER VARYING(10),

        		telefon_komorkowy CHARACTER VARYING(10),

        		adres_email CHARACTER VARYING(50),

        		adres_korespondencyjny INTEGER,

        		CONSTRAINT kontakty_2_fk FOREIGN KEY (nr_albumu) REFERENCES roznosci.studenci_2 (nr_albumu) ON DELETE CASCADE
        );

        CREATE TABLE roznosci.kontakty_3 (

        		nr_albumu CHARACTER VARYING(6),

        		telefon_domowy CHARACTER VARYING(10),

        		telefon_komorkowy CHARACTER VARYING(10),

        		adres_email CHARACTER VARYING(50),

        		adres_korespondencyjny INTEGER,

        		CONSTRAINT kontakty_3_fk FOREIGN KEY (nr_albumu) REFERENCES roznosci.studenci_3 (nr_albumu) ON UPDATE CASCADE
        );

      8. Odraczanie sprawdzenia więzów integralności z wykorzystaniem klauzuli DEFERRABLE.

        Klauzula DEFERRABLE pozwala odroczyć moment sprawdzenia integralności danych wprowadzanych do tabeli. Klauzula DEFERRABLE definiowana jest wraz z jedną poniższych warunków:

        • INITIALLY DEFERRED;
        • INITIALLY IMMEDIATE.

        Składnia:

        CREATE TABLE nazwa_schematu.nazwa_tabeli_powiązanej (

        		nazwa_kolumny_tab_powiązanej typ_danych,

        		...

        		nazwa_kolumny_tab_powiązanej typ_danych,

        		CONSTRAINT nazwa_ograniczenia FOREIGN KEY (nazwa_kolumny,...,nazwa_kolumny) REFERENCES nazwa_schematu.nazwa_tabeli_głównej (nazwa_kolumny_tab_głównej,...,nazwa_kolumny_tab_głównej)

        		[NOT DEFERRABLE | DEFERRABLE [INITIALLY DEFERRED | INITIALLY IMMEDIATE]]
        );

        Przykłady:

        • Ograniczenie NOT DEFFERABLE: Not-Deferrable.SQL
        • Ograniczenie DEFFERABLE INITIALLY DEFFERED: Deferrable-Initially-Deferred.sql
        • Ograniczenie DEFFERABLE INITIALLY IMMEDIATE: Deferrable-Initially-Immediate.sql

    1. Wprowadzenie.

      2. Język PL/pgSQL jest rozszerzeniem języka SQL wprowadzającym elementy programowania proceduralnego.

      Język PL/pgSQL obejmuje oraz umożliwia wykorzystanie:

      • zmienne, stałe,
      • wyrażenia,
      • struktury kontrolne (instrukcje warunkowe, pętle),

      W języku PL/pgSQL definiuje i tworzy się:

      • funkcje,
      • funkcje/procedury wyzwalane,
      • bloki obliczeniowe,
      • kursory.

      Składnia:

      1. blok programu PL/pgSQL:

        [<<etykieta>>]
        [DECLARE
        deklaracje]
        BEGIN
        polecenia
        END [etykieta];

      2. funkcja PL/PgSQL:

        CREATE FUNCTION schemat.funkcja(argumenty) RETURNS typ_danych AS $FUNKCJA$
        DECLARE
        zmienna_1 [CONSTANT] typ_danych [COLLATE sortowanie/porównanie] [NOT NULL] [{DEFAULT | :=} wartość];
        (...)
        zmienna_n [CONSTANT] typ_danych [COLLATE sortowanie/porównanie] [NOT NULL] [{DEFAULT | :=} wartość];

        polecenia;
        RETURN rezultat;
        END;

      a) Zmienne, stałe typów prostych.

      zmienna [CONSTANT] typ_danych [COLLATE sortowanie/porównanie] [NOT NULL] [{DEFAULT | :=} wartość];

      zmienna schemat.tabela%ROWTYPE;

Natomiast w przypadku typu złożonego RECORD struktura zmiennej tworzona jest w momencie przypisywania wartości i w związku z tym może ulegać dynamicznym zmianom w trakcie wykonywania.

      CREATE OR REPLACE FUNCTION roznosci.sprawdz_email(nr_albumu CHARACTER VARYING) RETURNS BOOLEAN AS
      $BODY$

      DECLARE
      email VARCHAR(256) := '';
      znak VARCHAR := '';
      kod INTEGER;
      i INTEGER := 1;
      dlugosc INTEGER := 0;

      BEGIN

      SELECT lower(email) INTO email FROM dziekanat.osobiste o WHERE o.nr_albumu = nr_albumu;

      dlugosc := length(email);

      -- Sprawdzenie czy występuje '@' oraz '.'
      IF strpos(email,'@') = 0 OR strpos(email,'.') = 0 THEN
      RAISE NOTICE 'Adres email jest niepoprawny.';
      RETURN FALSE;
      END IF;

      -- Sprawdzenie czy nie występują niedozwolone pozycje z '@' oraz '.'
      IF strpos(email,'.@') <> 0 OR strpos(email,'@.') <> 0 OR strpos(email,'..') <> 0 THEN
      RAISE NOTICE 'Adres email jest niepoprawny.';
      RETURN FALSE;
      END IF;

      -- Sprawdzenie czy występują tylko dozwolone znaki
      LOOP
      znak := substring(email, i, 1);
      kod := ascii(znak);
      IF NOT kod BETWEEN ascii('a') AND ascii('z') AND kod = ascii('_') AND kod = ascii('.') AND kod = ascii('@') THEN
      RAISE NOTICE 'Adres email zawiera nie prawidłowy znak: %', znak;
      RETURN FALSE;
      END IF;
      i := i+1;
      IF i > dlugosc THEN
      RETURN TRUE;
      END IF;
      END LOOP;
      END;
      $BODY$
      LANGUAGE 'plpgsql';

    1. Wprowadzenie

      2. Wyzwalacz (ang. trigger) jest obiektem bazy danych, który umożliwia wywołanie oraz wykonanie powiązanej procedury wyzwalanej w przypadku wystąpienia określonych zdarzeń w bazie danych.

      Wyzwalacz może zostać "aktywowany" w wyniku następujących zdarzeń:

      • operacji modyfikacji danych INSERT, UPDATE, DELETE dotyczącej tabeli/widoku;
      • operacji modyfikacji obiektu schematu bazy danych (np. tabeli);
      • akcji logowania/wylogowania użytkownika bazy danych (np. w ORACLE);

      Procedura wyzwalana (ang. trigger procedure) jest to funkcja bezargumentowa zapisana w języku proceduralnym/obiektowym wykorzystywana przez wyzwalacze (ang. trigger). Procedury wyzwalane tworzone są w języku/językach wspieranych przez SZBD, np. PL/pgSQL, PL/Tcl, PL/Perl, PL/Python, PL/SQL, C.

      Wyzwalacze wraz z procedurami wyzwalanymi są wykorzystywane w celu:

      • śledzenia operacji wykonywanych w bazie danych;
      • śledzenia zdarzeń systemowych bazy danych;
      • wymuszania więzów integralności referencyjnej;
        • które nie są możliwe do zdefiniowania deklaratywnie;
        • które są zdefiniowane w węzłach rozproszonej bazy danych;
      • realizacji złożonych reguł logiki biznesowej;
      • realizacji złożonych zasad polityki bezpieczeństwa;
      • uzupełniania tabel/widoków wartościami domyślnymi wg określonych reguł;

      Wyzwalacze dzielimy:

      • ze względu na sposób wykonania:
        • wykonywane dla instrukcji (ang. per-statement, statement-level) tzn. wyzwalacz jest wywoływany dokładnie jednokrotnie przy wykonywaniu określonej instrukcji;
        • wykonywane dla rekordu (ang. per-row, row-level) tzn. wyzwalacz jest wywoływany wielokrotnie (0, 1,...,n razy) dla każdego rekordu na który wpływa (np. modyfikuje) instrukcja uruchamiająca wyzwalacz;
      • ze względu na czas wykonania:
        • wykonywane przed (ang. before trigger) tzn. wyzwalacz jest wykonywany przed instrukcją (w przypadku statement-level triggers) lub przed modyfikacją rekordu (w przypadku row-level triggers);
        • wykonywane po (ang. after trigger) tzn. wyzwalacz wykonywany po instrukcji (w przypadku statement-level triggers) lub po modyfikacji rekordu (w przypadku row-level triggers);

      Dozwolonymi wartościmi zwracanymi przez wyzwalacze są:

      • dla wyzwalaczy typu statement-level before tylko wartość NULL;
      • dla wyzwalaczy typu row-level after dowolna wartość (zwrócona wartość jest ignorowana);
      • dla wyzwalaczy typu row-level before:
        • NULL w celu pominięcia operacji na bieżącym rekordzie;
        • rekord (wartość typu HeapTupple) który zostanie użyty w operacji INSERT lub UPDATE;

      W praktyce wyzwalacze row-level before są wykorzystywane do sprawdzenia lub modyfikacji danych, które są dodawane lub aktualizowane w bazie danych, natomiast wyzwalacze row-level after do propagowania zmian w tabelach bazy danych tak, aby została zachowana spójność danych.

    TEMAT : Transakcje
    1. Wprowadzenie

      Pojęcie ACID definiuje zasadnicze własności transakcji w bazie danych:

      • ATOMICITY (atomiczność, niepodzielność) określa, że cała sekwencja operacji w obrębie transakcji musi zostać zakończona pomyślnie albo musi zostać anulowana. Transakcja nie może zakończyć się częściowo pomyślnie;
      • CONSISTENCY (spójność) określa, że wykonanie transakcji, albo utworzy nowy stan danych w którym wszystkie zmiany zostały wprowadzone gwarantując ich spójność, albo, w przypadku niepowodzenia, zostanie przywrócony stan danych istniejący przed transakcją.
      • ISOLATION (izolacja) określa możliwość wykonywania operacji w obrębie transakcji w pełnej izolacji (niezależności) od pozostałych operacji przeprowadzanych w bazie danych. Operacje (np. modyfikacji danych) przeprowadzane w ramach transakcji nie są widoczne dla innych transakcji, aż do momentu zatwierdzenia transakcji.
      • DURABILITY (trwałość) określa, że zmiany w danych w ramach zatwierdzonej transakcji są trwałe, tzn. nie zostaną utracone w wyniku okoliczności zewnętrznych (np. awarii).

      Współbieżne przypadki odczytów w bazie:

      • Nieważne odczyty (ang. DIRTY READS)
      • Odczyty niepowtarzalne (ang. NON-REPEATABLE READS)
      • Odczyty fantomowe (ang. PHANTOM READS)

      Nieważne odczyty

      Nieważny odczyt występuje gdy transakcja może odczytać dane, których zmiana wykonana w ramach innej transakcji nie została jeszcze zatwierdzona/odrzucona. Przedstawiając przypadek nieważnego odczytu precyzyjniej, jeżeli określona transakcja zmieni wartość danych a inna tą zmienioną wartość danych odczyta, a następnie wprowadzone w ramach transakcji zmiany zostaną odrzucone (wycofanie transakcji) spowoduje to, że wartości danych staną się nieważne ("brudne").

      Przykład

      Załóżmy, że dana jest tabela Statystyki (tab. 1) w ramach której wykonywane są dwie transakcje T1 oraz T2. Transakcja T2 wykonywana jest na poziomie izolacji READ UNCOMMITTED, zatem dopuszczone są w transakcji tzw. nieważne odczyty.

      ID Fakultet Zapisy
      #1 Kryptografia 5
      #2 Systemy ekspertowe 8
      #3 UML 11

      Tab. 1 Statystyki

      1. Rozpoczęcie wykonywania transakcji T1 oraz T2;
      2. T1 aktualizuje wartość Zapisy w wierszu #1 na 12;
      3. T2 odczytuje nie zatwierdzoną przez T1 wartość z wiersza #1;
      4. T1 wycofuje wprowadzone zmiany, zatem w wierszu #1 wartość Zapisy ponownie wynosi 5;
      5. T2 wciąż posiada i operuje na wartości Zapisy równej 12, zatem wartości nieważnej ("brudnej");

      Odczyty niepowtarzalne

      Odczyt niepowtarzalny występuje gdy dane (wiersz) które nie zostały zaktualizowane podczas transakcji są wewnątrz niej odczytywane dwukrotnie z różnymi rezultatami.

      Przykład

      Załóżmy, że dana jest tabela Statystyki (tab. 1) w ramach której wykonywane są dwie transakcje T1 oraz T2. Transakcja T2 wykonywana jest na poziomie izolacji READ COMMITTED, zatem dopuszczone są w transakcji tzw. niepowtarzalne odczyty.

      1. Rozpoczęcie wykonywania transakcji T1 oraz T2;
      2. T2 odcztuje wartość Zapisy wiersza #1 wynoszącą 5;
      3. T1 aktualizuje wartość Zapisy w wierszu #1 na 12 i zatwierdza zmiany;
      4. T2 ponownie odczytuje wartość Zapisy wiersza #1, która aktualnie wynosi 12, ponieważ poziom izolacji transakcji T2 oznacza, że zmiany wprowadzane w ramach innych transakcji są dla transakcji T2 widoczne, nawet jeśli T2 nie została zakończona (zatwierdzenie/odrzucenie zmian).

      Odczyty fantomowe

      Odczyt fantomowy występuje, gdy transakcja odczytuje dane (wiersz) wstawione w obrębie innej transakcji, która nie została zatwierdzona. Jeżeli określona transakcja wstawia wiersz do tabeli, w czasie gdy inna transakcja z tej tabeli dane odczytuje, może wystąpić sytuacja, że zostanie odczytany nowo dodany wiersz, nawet jeśli transakcja (dodająca wiersz) zostanie wycofana.

      Przykład

      Załóżmy, że dana jest tabela Statystyki (tab. 1) w ramach której wykonywane są dwie transakcje T1 oraz T2. Transakcja T2 wykonywana jest na poziomie izolacji REPEATABLE READS, zatem dopuszczone są w transakcji tzw. odczyty fantomowe.

      1. Rozpoczęcie wykonywania transakcji T1 oraz T2;
      2. T2 wykonuje zapytanie SELECT * FROM Statystyki WHERE Zapisy <= 5 i odczytuje jeden wiersz #1;
      3. T1 wstawia do tabeli nowy wiersz ('#4', 'Sieci neuronowe', 3);
      4. T2 ponownie wykonuje zapytanie SELECT * FROM Statystyki WHERE Zapisy <= 5 i odczytuje dwa wiersze #1 i #4;
      5. T1 wycofuje zmiany wprowadzone w ramach transakcji, zatem wiersz #4 aktualnie w tabeli nie istnieje.
      6. T2 z powodzeniem zatwierdza zmiany danych bazując na dodatkowym fantomowym wierszu w tabeli Statystyki;
